### Part 1 - Naive simulation (`NaivePhilosophers.java`)

Each philosopher is a thread. The 5 chopsticks are `ReentrantLock` objects shared between neighbors. A philosopher grabs the left chopstick, then the right, eats, and puts them back.

This satisfies the basic requirement (two philosophers never hold the same chopstick at the same time), but it can deadlock: if every philosopher grabs their left chopstick simultaneously, they all block waiting for the right one forever.

The program has a 5-second timeout: if threads are still alive after that, it reports a deadlock and interrupts them.

### Part 2 - Deadlock-free (`DeadlockFreePhilosophers.java`)

Deadlock requires circular wait. We break it with resource ordering: every philosopher always picks up the lower-numbered chopstick first.

Philosopher 4 sits between chopsticks 4 and 0, so with the naive approach it would grab 4 then 0. With ordering it grabs 0 then 4, same as philosopher 0. This means they can't both hold one chopstick and wait on the other in a cycle. No circular wait, no deadlock.

Since deadlock is no longer possible, we can `join()` without a timeout and drop the deadlock-detection logic.

### Part 3 - Starvation-free (`StarvationFreePhilosophers.java`)

Resource ordering prevents deadlock but doesn't prevent starvation - a philosopher between two fast eaters could theoretically wait forever. For example, philosopher 2 sits between 1 and 3. If both 1 and 3 are fast eaters, they could keep alternating: every time Phil 2 wakes up and tries to grab the lock, one of its neighbors beats it and starts eating first. Phil 2 never gets a turn.

We switch to a different design that combines two ideas.

**A single fair lock as a coordinator.** Instead of locking individual chopsticks, we use one `ReentrantLock(true)` shared by everyone. The `true` makes it fair: the lock maintains a FIFO queue, so when multiple threads are waiting, the one that's been waiting longest goes first. No thread can be skipped indefinitely.

**Per-philosopher `Condition` variables.** A `Condition` (created via `lock.newCondition()`) lets a thread atomically release the lock and go to sleep, then re-acquire the lock when woken. Each philosopher gets their own `Condition`, so when philosopher 2 finishes eating it only wakes its neighbors (1 and 3) via `cond[1].signal()` and `cond[3].signal()`, not everyone.

A philosopher's loop works like this:
1. Acquire the lock.
2. While either neighbor is eating, call `cond[id].await()`, which releases the lock and sleeps.
3. When woken (by a neighbor finishing), re-acquire the lock and check again.
4. Once neither neighbor is eating, set `eating[id] = true` and release the lock.
5. Eat.
6. Acquire the lock, set `eating[id] = false`, signal both neighbors, release the lock.

The `await()` call is always inside a `while` loop (not `if`) because a thread can be woken spuriously, or the condition might no longer hold by the time it re-acquires the lock.

Starvation freedom comes from the fair lock: once a philosopher enters the FIFO queue, there are at most N-1 threads ahead of it. It will eventually get served.

### Part 4 - Starvation-free for any n (`StarvationFreeNPhilosophers.java`)

Same idea as Part 3. The only change is that `N` is no longer `final` - it can be set from the command line (defaults to 5). The shared state (`lock`, `cond`, `eating`) is initialized in `main` instead of at declaration time so it can use the runtime value of `N`. The `pickUp`, `putDown`, `leftOf`, and `rightOf` methods stay the same.

Usage: `java StarvationFreeNPhilosophers 12` runs with 12 philosophers.
