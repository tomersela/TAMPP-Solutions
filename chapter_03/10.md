# Execution showing that the linearization point for enq() cannot occur at line 15

- Thread A calls `enq(1)` runs line 15 (`int i = tail.getAndIncrement();`).
  At the point the local value of `i` is 0.
- The OS preempts thread A and schedules thread B, which calls `enq(2)` - runs both line 15 and 16 (`items[i].set(x);`). At this point `item[1]` is set to 2
- Now thread C calls `deq()` and gets 2.
- Thrad A resumes, and run line 16. Now `item[0]` is set to 1.
- Thread C is scheduled again and calls `deq(1)` and gets 1.

Even though thread A executed line 15 before thread B executed line 15, the first dequeued item was the one thread B enqueued. Therefore line 15 wasn't a linearization point for this execution.

# Execution showing that the linearization point for enq() cannot occur at line 16

- Thread A:
    - Calls `enq(0)` and returns. Now `item[0]` equals to 0.
    - It then call `deq()`. At this point `item[0]` is null (because of the call to `deq()`), and `tail` equals 1.
    - Calls `enq(1)`, run line 15 and gets preempted before running line 16. At this point `tail` is equal to 2.
- Thread B calls `enq(2)`. It run line 15 and gets preempted. Now `tail` equals 3.
- Thread C:
    - calls `deq()`, and enters the for loop. `tail` is 3, so there will be 3 iterations at most. All the first three items are null.
    - The loop advances 2 iterations, since the first two items are null. `i` is now 2.
    - The thread get preempted before running the third iteration.
- Thread A continuous and run line 16. Now `item[1]` is 1.
- Thread B continuous and run line 16. Now `item[2]` is 2.
- Thread C continuous and run the third iteration (`i` equals 2). Since `item[2]` is 2, it returns 2.
- Thread C calls `deq()` again and now returns 1.


Even though thread A ran line 16 before thread B, it was B's value that got dequeued first.  Therefore line 16 wasn't a linearization point for this execution.

# Does this mean enq() is not linearizable?
No. Linearizability is a property of the observable history (invocations and responses), not of where inside the code a linearization point falls. In both executions above, the concurrent `enq()` calls overlap in time, so they can be ordered either way. Since there exists a total order consistent with real-time and the sequential FIFO specification, linearizability holds — even though `enq()` has no single fixed linearization point in its own code.

That said, `enq()` is linearizable in the context of how `deq()` is implemented. A different `deq()` implementation could break the linearizability of the whole object, even with the same `enq()` code.
