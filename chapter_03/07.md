```java
1:  class IQueue<T> {
2:      AtomicInteger head = new AtomicInteger(0);
3:      AtomicInteger tail = new AtomicInteger(0);
4:      T[] items = (T[]) new Object[Integer.MAX_VALUE];
5:      public void enq(T x) {
6:          int slot;
7:          do {
8:              slot = tail.get();
9:          } while (!tail.compareAndSet(slot, slot+1));
10:         items[slot] = x;
11:     }
12:     public T deq() throws EmptyException {
13:         T value;
14:         int slot;
15:         do {
16:             slot = head.get();
17:             value = items[slot];
18:             if (value == null)
19:                 throw new EmptyException();
20:         } while (!head.compareAndSet(slot, slot+1));
21:         return value;
22:     }
23: }
```

The following scheduling shows non-linearizable behavior:

```
A: ---[ enq(1)                ]-------------------------
B: ----------[ enq(2) ]--[ deq() == EmptyException]----
```

- Thread A invokes enq(1).
- The OS now preempts A just after the do-while loop and before line 10 (`items[slot] = x;`).
  At this point the local variable `slot` equals 0 and `items[0]` is still `null`.
- The OS schedules thread B, which invokes enq(2). It runs to completion, and now `items[1]` equals 2 and `tail` equals 2.
- Then B calls deq(), which reads `head` (== 0), then reads `items[0]` and gets `null`, then the condition at line 18 is true and therefore throws `EmptyException`.
- The OS resumes thread A, which now sets `items[0]` to 1.

Thread B completed `enq(2)` before starting `deq()`. So any linearizable total order should respect `B: enq(2) → B: deq()`. But it's impossible for a valid FIFO to have an item enqueued and then yield an empty state when dequeued.
