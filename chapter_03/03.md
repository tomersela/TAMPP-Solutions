# Quiescently consistent but not sequentially consistent

This example describes two threads interacting with register r concurrently in the following real time order:
```
<r.write(0) A>, <r: Ok(void)>, <r.read() B>, <r.write(1) A>, <r: Ok(1), B>, <r.read() B>, <r: Ok(void) A>, <r: Ok(0) B>
```

Visually:
```
A: -----[ write(r = 0)]------[ write(r = 1) ]---------------
B: -----------------------[read(r == 1)]--[read(r == 0)]----
```

In quiescent consistency, we can reorder {`write(r = 1)`, `read(r == 1)`, `read(r == 0)`}, but we should keep `write(r = 0)` before other events. So a possible total order to explain QC:

- A: write(r = 0)
- B: read(r == 0)
- A: write(r = 1)
- B: read(r == 1)

This example isn't sequentially consistent since we're not allowed to reorder thread B calls.
Any total order in SC needs to preserve per thread ordering: `A: write(r = 0) → A: write(r = 1)` and `B: read(r == 1) → B: read(r == 0)`.
Since thread B read r and got 1, and the only write to r that sets it to 1 is A's `write(r = 1)`. Therefore, it has to be that `A: write(r = 1) → B: read(r == 1)`. Now:

`A: write(r = 0) → A: write(r = 1) → B: read(r == 1) → B: read(r == 0)`.

We see that r was changed from 1 to 0 without any write in between, which doesn't make sense.


# Sequentially consistent but not quiescently consistent
The following real-time ordering can be explained with sequential consistency:
```
<r.write(0) A>, <r.read() B>, <r: Ok(void) A>, <r: Ok(1), B>, ... Quiescent period ..., <r.write(1) A>, <r: Ok(void) A>
```

Visually:
```
A: -----[ write(r = 0)]--------------[ write(r = 1) ]-------
B: ---------------[read(r == 1)]----------------------------
```

In sequential consistency we can explain this by saying that B called read(r == 1) last:

- A: write(r = 0)
- A: write(r = 1)
- B: read(r == 1)

This is ok since we still preserve the order of events per thread in the total ordering.

But this scheduling isn't quiescently consistent!
In QC, method calls separated by a period of quiescence must appear in real-time order in any witness. Here B's read completes, then there is a quiescent period, and only after that does A's `write(r = 1)` run. So any QC witness must have **B's read before `write(r = 1)`**. But a read that returns **1** needs **`write(r = 1)` before that read** in any register-consistent witness. Those two constraints contradict each other, so the history is not quiescently consistent.
