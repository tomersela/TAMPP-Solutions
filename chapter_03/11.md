# Execution showing that the linearization point for push() cannot occur at line 11

- Thread A calls `push(a)`, runs line 11 (`int i = top.getAndIncrement();`). A gets `i = 0`. A is preempted before line 12.
- Thread B calls `push(b)`, runs both line 11 and line 12. B gets `i = 1`, then stores `b` at `items[1]`.
- Thread C calls `pop()`. It reads `top = 2`, scans descending: `items[1]` has `b`, swaps it out and returns `b`.
- Thread A resumes and runs line 12, storing `a` at `items[0]`.
- Thread C calls `pop()` again and returns `a`.

Thread A executed line 11 before thread B, so if line 11 were the linearization point, `push(a)` would be linearized first and `pop()` should return `a` first (LIFO). But `pop()` returned `b` first. Therefore line 11 is not a fixed linearization point.

# Execution showing that the linearization point for push() cannot occur at line 12

- Thread A calls `push(a)`, runs line 11 (`i = 0`). Preempted before line 12.
- Thread B calls `push(b)`, runs line 11 (`i = 1`). Preempted before line 12.
- Thread A resumes, runs line 12: `items[0] = a`.
- Thread B resumes, runs line 12: `items[1] = b`.
- Thread C calls `pop()`. Reads `top = 2`, scans descending: finds `b` at `items[1]`, returns `b`.
- Thread C calls `pop()` again, returns `a`.

Thread A executed line 12 before thread B, so if line 12 were the linearization point, `push(a)` would be linearized first. In a LIFO stack, `pop()` should then return `b` first (last in, first out) — and it does. So this execution alone doesn't separate. We need a different one:

- Thread A calls `push(a)`, runs line 11 (`i = 0`). Preempted before line 12.
- Thread B calls `push(b)`, runs line 11 (`i = 1`) and line 12: `items[1] = b`.
- Thread C calls `pop()`. Reads `top = 2`, scans descending: finds `b` at `items[1]`, returns `b`.
- Thread A resumes, runs line 12: `items[0] = a`.
- Thread C calls `pop()` again, returns `a`.

If line 12 were the linearization point, `push(b)` is linearized first (B ran line 12 first), then `push(a)`. LIFO order means `pop()` should return `a` first, but it returned `b` first. Therefore line 12 is not a fixed linearization point.

# Does this mean push() is not linearizable?
No. Linearizability is a property of the observable history (invocations and responses), not of where inside the code a linearization point falls. In both executions above, the concurrent `push()` calls overlap in time, so they can be ordered either way. Since there exists a total order consistent with real-time and the sequential stack specification, linearizability holds — even though `push()` has no single fixed linearization point in its own code.

As with the HWQueue in Exercise 3.10, `push()` is linearizable in the context of how `pop()` is implemented. A different `pop()` implementation could break the linearizability of the whole object.
