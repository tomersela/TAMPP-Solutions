**Counterexample.** Start in a valid sequential state, for example:

- thread `0`: `(0, 010)`
- thread `1`: `(1, 001)`
- thread `2`: `(2, 100)`

(so labels are acyclic / totally ordered under sequential use). A concurrent interleaving of `scan` / `label` can leave the system in:

- thread `0`: `(0, 001)`
- thread `1`: `(1, 011)`
- thread `2`: `(2, 011)`

Under the $T_3$ rule: $(0,001) \succ (1,011)$ because $0<1$ and `b[1] = c[0]` is `0 = 0`; $(1,011) \succ (2,011)$ because $1<2$ and `b[2] = c[1]` is `1 = 1`; $(2,011) \succ (0,001)$ because $2>0$ and `b[0] ≠ c[2]` is `0 ≠ 1`. So `later` has a **3-cycle** on these three labels, hence they are not totally ordered (thread ids only break ties when two labels match; here the bit strings differ).

Hence sequential $T_3$ is not a correct concurrent timestamp system for three threads.
