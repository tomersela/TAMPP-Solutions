```java
class Flaky implements Lock {
    private int turn;
    private boolean busy = false;
    public void lock() {
        int me = ThreadID.get();
        do {
            do {
                turn = me;
            } while (busy);
            busy = true;
        } while (turn != me);
    }
    public void unlock() {
        busy = false;
    }
}
```

### Does this protocol satisfy mutual exclusion
Yes.

**Lemma.** The Flaky lock algorithm satisfies mutual exclusion.<br/>
Proof: Suppose not. Then there exist two distinct threads A and B such that both enter the critical section.

For a thread X to enter the critical section, it must exit the outer do-while loop, so it must observe

read_X(turn == X)

after executing

write_X(busy = true).

Thus, if A and B both enter, then we must have:

write_A(busy = true) → read_A(turn == A) → CS_A

write_B(busy = true) → read_B(turn == B) → CS_B

Now consider the last writes to busy = true by A and B before these critical-section entries. Since the precedence order on events is total, one of them is later. Without loss of generality, suppose

write_A(busy = true) → write_B(busy = true).

After write_A(busy = true), the variable busy remains true until some thread executes unlock(). Therefore, after A executes write_A(busy = true), no thread can pass line 9 and execute line 10 unless some thread first leaves the critical section and sets busy = false.

But B executes write_B(busy = true) before entering its critical section, so B must have passed line 9 after observing busy == false. That is impossible unless some thread had already executed unlock(), contradicting the assumption that A and B are both in the critical section.

Hence two threads cannot be in the critical section simultaneously. Therefore the algorithm satisfies mutual exclusion.

### Is this protocol starvation-free?
No. A thread may get stuck in the do-while loop that check for the `busy` variable if other threads are fast enough to mark this variable to true again.

### is this protocol deadlock-free?
No.

Consider two threads, A and B.

- A executes turn = A.
- A reads busy == false, so it exits the inner loop.
- A executes busy = true.
- B executes turn = B.
- B reads busy == true, so B stays in the inner loop.
- A evaluates the outer-loop condition and reads turn == B, so turn != A is true. Therefore A does not enter the critical section and starts the outer loop again.
- A now goes back to the inner loop, but busy is still true, so A also spins there forever.

Now both threads are stuck in the inner loop because busy == true, and none of them gets into the critical section.
