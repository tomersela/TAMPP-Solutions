### Overview

The exercise asks us to implement the `Timestamp` interface from Fig. 2.10 using unbounded labels, and then rewrite the Bakery lock from Fig. 2.9 using that implementation.

The original Bakery lock uses raw integer labels: each thread picks a label larger than all current labels, then spins until no earlier-labeled thread is trying to enter. The idea here is to formalize that labeling as a `TimestampSystem`.

The Bakery algorithm is notable for requiring only reads and writes - no CAS, no test-and-set, no hardware atomics. The `label` method just scans all current timestamps, finds the max, and writes max + 1. That's it.

### How the pieces fit together

**`Timestamp` interface** - represents a single label. The `compare` method returns true if this timestamp is earlier (smaller) than the other. This is the ordering the Bakery lock uses to decide who goes first.

**`UnboundedTimestamp`** - implements `Timestamp` by wrapping a `long`. Two timestamps are compared by their long values. "Unbounded" means the labels grow without limit (in practice, a 64-bit long won't overflow).

**The `TimestampSystem` interface** provides two operations:

- `scan()` returns a snapshot of all threads' current timestamps.
- `label(timestamp, i)` assigns a new timestamp to thread $i$ that is guaranteed to be larger than all existing ones.

**`UnboundedTimestampSystem`** implements this by scanning all current timestamps, finding the maximum, and assigning max + 1. This is a direct translation of the pseudocode's `max(label[0], ..., label[n-1]) + 1`. It only uses reads (the scan) and a single write (setting the new value); no atomic read-modify-write operations.

**`BakeryLock`** - the lock itself. The `lock` method:
1. Sets `flag[i] = true` (thread $i$ wants in).
2. Calls `ts.label(label[i], i)` to get a fresh timestamp via scan-and-max.
3. Spins on every other thread $k$: waits while $k$ has its flag set and $(label[k], k) < (label[i], i)$ lexicographically. Smaller label wins; if labels are equal, smaller thread id wins.

The `unlock` method just clears the flag.

**Note on Java memory visibility:** the textbook assumes sequential consistency, but Java doesn't provide that by default. Array elements aren't volatile, so without explicit memory ordering the flag writes might not be visible across threads. We use `VarHandle` to do volatile reads and writes on the `flag` array elements. This isn't adding atomic RMW - it's just ensuring that plain reads and writes are visible across threads, which is what the textbook assumes for free.

### Test (`BakeryTest.java`)

5 threads each increment a shared counter 20,000 times through the Bakery lock. If mutual exclusion holds, the final count is exactly 100,000. Any violation would show up as a lower count since increments of a non-atomic `int` would get lost under concurrent access.

#### Example run

```
$ javac *.java && java BakeryTest
Bakery Lock test: 5 threads, 20000 increments each
Expected final count: 100000
Actual final count:   100000
PASS -- mutual exclusion held.
Time: 50 ms
```
