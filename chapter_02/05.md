```java
public void unlock() {
    int i = ThreadID.get();
    flag[i] = false;
    int j = 1 - i;
    while (flag[j] == true) {}
}
```

`lock()` is standard two-thread Peterson (`flag[i] = true`, `victim = i`, `while (flag[j] && victim == i) {}`). Assume the exercise’s finite-delay model: each thread that still has steps to run gets scheduled again after finite time (fairness). Starvation-freedom is proved in that model. Dropping it, an OS could starve `i` between `flag[i] = false` and the following `while`, or delay `i`’s reads inside that loop arbitrarily long, even if other threads look “fairly” scheduled.

### Deadlock-free

Yes.

If thread `i` blocks in `while (flag[j])` at the end of `unlock()`, then `flag[i]` is already false. Thread `j` cannot be stuck on Peterson’s wait `while (flag[i] && victim == j)`, since that needs `flag[i] == true`. With finite scheduling, `j` eventually either clears `flag[j]` in its own `unlock()` after the critical section or already has `flag[j] == false`, so `i`’s wait terminates. No cycle pairs `i` waiting on `flag[j]` with `j` waiting on a predicate that requires `flag[i] == true`.

### Starvation-free

Yes.

While `i` is in the trailing `while (flag[j])`, `flag[i]` is false, so `j`’s Peterson wait does not block on `flag[i]`. Between `flag[i] = false` and evaluating `while (flag[j])`, `j` may run many complete `lock`/`unlock` cycles under a greedy schedule; under the model above, `i` still reaches that `while` eventually. If `flag[j]` is then true, `i` spins until the next `flag[j] = false` in `j`’s `unlock()` (finite critical section, same scheduling assumption). Symmetric for `j`.
