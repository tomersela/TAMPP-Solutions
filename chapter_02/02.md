We need a doorway because first-come-first-served requires a precise moment when a thread is considered to have joined the competition for the lock. The first instruction of `lock()` is not a reliable arrival marker: depending on whether it is a read or a write, and whether it targets the same shared location or different ones, other threads generally cannot recover the global order in which those first instructions executed, so the algorithm cannot use that order to enforce FCFS.

### Read of a separate location

No shared variable is updated, so other threads see nothing attributable to this arrival. Even if thread A’s read logically happens before B’s, that fact is not recorded in memory for the protocol to use.

### Read of the same location

Same issue - reads do not change memory, so the sequence of first reads leaves no footprint. Later code cannot tell who read first.

### Write to a separate location

Each thread writes its own word. After both writes, memory records the final values but does not, by itself, reveal which write happened first, so first-instruction order is not recoverable from that evidence alone.

### Write to the same location

The second write overwrites the first. Evidence of which thread wrote first may disappear entirely, so again the first-instruction order is not reliably recoverable.

### Why a doorway fixes the gap

A doorway is a bounded prefix of `lock()` whose end is fixed by the algorithm so that “A finished its doorway before B started its doorway” is a predicate the implementation can use (via later reads and writes). That supports FCFS and related fairness definitions; ordering by the very first instruction does not.
