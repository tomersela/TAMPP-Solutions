### Idea

We want a sequential timestamp system for `n` threads using only `n * 2^n` possible labels.

Let a label be a pair

$$
(i, b)
$$

where:

- $i$ is the thread id, $i \in \lbrace 0,1,\dots,n-1 \rbrace$
- `b` is an `n`-bit string

So the total number of possible labels is:

$$
n \cdot 2^n
$$

because there are `n` choices for the thread id and `2^n` choices for the bit string.


### How to compare two labels

Suppose we want to compare two labels:

$$
(i, b) \quad \text{and} \quad (j, c), \qquad i \ne j
$$

We compare them using only:

- bit `b[j]` from the first label
- bit `c[i]` from the second label

The rule is:

- if $i < j$, then $(i,b)$ is later than $(j,c)$ iff

$$
b[j] = c[i]
$$

- if $i > j$, then $(i,b)$ is later than $(j,c)$ iff

$$
b[j] \ne c[i]
$$

So comparison depends only on the two labels themselves, as required.


### How thread `i` chooses a new label

When thread `i` wants a new label, it scans all current labels.

For each other thread `j`, suppose thread `j` currently has label `(j, c)`.

Thread `i` chooses its own bit `b[j]` so that its new label `(i, b)` will be later than `(j, c)`.

So for each `j != i`:

- if $i < j$, choose

$$
b[j] = c[i]
$$

- if $i > j$, choose

$$
b[j] = 1 - c[i]
$$

The bit `b[i]` does not matter and can be chosen arbitrarily.

This guarantees that the new label `(i,b)` is later than every currently existing label.

Since the system is sequential, the scan and label operation happen atomically, so the thread is compared only against the labels it saw in that scan.


### Why this works

For each other thread `j`, thread `i` explicitly chooses the bit `b[j]` so that the pairwise comparison rule makes `(i,b)` later than thread `j`’s current label.

Therefore the new label is later than all current labels.

Also, once labels are chosen, any two labels can be compared without looking at the rest of the system: only the two thread ids and the two relevant bits are needed.

Thus this is a valid sequential timestamp system.


## Intuition with `n = 3`

For `n = 3`, a label has the form:

$$
(i, xyz)
$$

where `i` is the thread id and `xyz` is a 3-bit string.

Suppose the current labels are:

- thread `0`: `(0, 010)`
- thread `1`: `(1, 100)`

Now thread `2` wants a new label later than both.

### Step 1: beat thread 0

Since `2 > 0`, thread `2` must choose its bit for thread `0` to be the opposite of thread `0`’s bit for thread `2`.

Thread `0` has label `010`, so its bit for thread `2` is `0`.

Therefore thread `2` chooses:

- its bit for thread `0` = `1`

### Step 2: beat thread 1

Since `2 > 1`, thread `2` must choose its bit for thread `1` to be the opposite of thread `1`’s bit for thread `2`.

Thread `1` has label `100`, so its bit for thread `2` is `0`.

Therefore thread `2` chooses:

- its bit for thread `1` = `1`

### Step 3: choose its own bit

The bit for itself does not matter, so choose anything, say `0`.

So thread `2` chooses the label:

$$
(2, 110)
$$

### Check it

Compare `(2,110)` with `(0,010)`:

- thread `2`’s bit for thread `0` is `1`
- thread `0`’s bit for thread `2` is `0`

Since `2 > 0`, `2` is later iff the bits are different. They are, so `(2,110)` is later.

Compare `(2,110)` with `(1,100)`:

- thread `2`’s bit for thread `1` is `1`
- thread `1`’s bit for thread `2` is `0`

Again, since `2 > 1`, `2` is later iff the bits are different. They are, so `(2,110)` is later.

So thread `2` successfully chooses a label later than both current labels.


### Final answer

A valid sequential timestamp system with only `n * 2^n` labels is obtained by representing each label as:

$$
(i, b)
$$

where `i` is the thread id and `b` is an `n`-bit string, and defining comparison and label selection exactly as above.
